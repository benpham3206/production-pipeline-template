## Design Document Phase and Anti-Test-Gaming Safeguards Added
**Context:** User asked how to ensure agents write *accurate* code vs. plausible/test-passing code, and requested a formal Design Document phase with objectives, constraints, criteria, and test philosophy.
**Decision:**
- Created `docs/DESIGN_DOC.md` establishing:
  - High-level objectives (functional + non-functional)
  - Detailed objectives (Salvatier decomposition)
  - Metrics (measurable, meaningful)
  - Constraints (the hard line — cross it, design is out)
  - Criteria (better-looks-like-this optimization)
  - Test-to-objective mapping (no orphan tests)
  - Test philosophy (purpose, method, results, confidence discussion)
  - Adversarial test review (agent must find loopholes before coding)
  - Ordered recommendations
- Created `docs/templates/design-doc.md` as the working template
- Created `docs/ANTI_TEST_GAMING.md` with 7 safeguards:
  1. Design Document Constraints (architecture, security, maintainability)
  2. Adversarial Test Review (agent tries to break tests)
  3. Black-Box Test Requirement (behavioral, not implementation-dependent)
  4. Property-Based and Fuzz Testing (where applicable)
  5. Constraint Audit After Coding (independent of test suite)
  6. Human Review for "Plausible but Wrong"
  7. Test Mutation Check (would code still be correct without this test?)
- Updated workflow to: **PRD → UI/UX → Test Definition → Design Doc → Code**
- Updated `AGENTS.md`, `docs/PROCESS.md`, `docs/TESTING.md`, `INDEX.md`, `health-check.sh`, `init-project.sh`
**Rationale:** Tests alone cannot prevent AI agents from writing plausible-but-wrong code. Constraints, adversarial reviews, and design intent create a multi-layer defense. The Design Document forces the agent to reason about *why* the design is trustworthy before optimizing for green tests.
**Trade-offs:** Significant upfront documentation. However, this is exactly where agentic coding saves time — by doing the thinking correctly before generating code.
**Consequences:** The pipeline now explicitly prevents "Goodhart's Law" in code generation. Passing tests is necessary but not sufficient. The code must satisfy design intent, constraints, and pass adversarial scrutiny.

## Clean Exit Audit for All Functions and Scripts
**Context:** User requested that all return, callback, and timeout functions be clean and exit cleanly/report back.
**Decision:**
- Audited all 5 shell scripts and all TypeScript source files for clean exits
- Added `set -euo pipefail` to all shell scripts:
  - `checkpoint.sh`
  - `health-check.sh`
  - `init-project.sh`
  - `pre-ship-audit.sh`
  - `recovery.sh`
- Added explicit `exit 0` at the end of every successful script path
- Fixed `checkpoint.sh` to fall back from `tac` to `tail -r` for macOS compatibility
- Fixed `checkpoint.sh` restore to use `"$TARGET/"*` (trailing slash) to ensure reliable content copying even with empty directories
- Verified all TypeScript functions have guaranteed return values on every code path
- Confirmed no async callbacks, Promises, setTimeout, or other potentially hanging operations exist in the source code
- Tested `checkpoint.sh` create, list, and restore; tested `recovery.sh` with and without backups; all exit with correct codes
**Rationale:** Scripts that fail silently in pipelines or hang on missing input are a common source of CI/debugging pain. `pipefail` ensures pipeline failures are caught. Explicit exit codes make script behavior predictable for callers.
**Trade-offs:** `set -u` requires that all variables be defined before use, which is slightly stricter but catches typos. `set -o pipefail` is bash-specific but available on macOS and all modern Linux distributions.
**Consequences:** Every script in the pipeline now exits cleanly with a well-defined status code, and no function can hang or leak pipeline failures.
